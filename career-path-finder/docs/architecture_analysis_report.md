# Architecture Implementation Analysis

> [!NOTE]
> This document analyzes the current state of the `career-path-finder` codebase against the designed system architecture. It breaks down what is genuinely implemented, what is merely mocked or entirely missing, and which components exist outside the original design scope.

## 1. Components Analysis

### 1.1 SkillsParser: The Ingestion Engine
**Status: Partially Implemented (Mocked / Basic Heuristics)**
*   **Implemented:** `github_live.py` successfully hits the public GitHub API to fetch a user's repositories and calculates the percentage of bytes per language. `parsers.py` includes a `DocumentParser` that extracts text from PDF resumes/transcripts up to 5 pages.
*   **Missing (Design Gap):** There are no LLMs utilized for semantic entity extraction or disambiguation. Instead, it relies on entirely hardcoded dictionaries (e.g., `LANGUAGE_TO_SKILLS` in `github_live.py` and `keywords` in `parsers.py`) to map keywords/languages directly to confidence scores.

### 1.2 WorldCompiler: The Institutional Reasoning Engine
**Status: Not Implemented**
*   **Implemented:** None of the core requirements.
*   **Missing (Design Gap):** There is no logic for ingesting academic course catalogs or syllabi. The system does not construct a canonical Directed Acyclic Graph (DAG) for prerequisites, corequisites, credit hours, or equivalency mapping.

### 1.3 PathFinder: The Backward-Chaining Algorithm
**Status: Partially Implemented (via Statistical Workaround & Mocks)**
*   **Implemented**: `jobhop_engine.py` implements a version of graph traversal (Dijkstra) to find optimal career routes. It operates on a statistical transition graph derived from a dataset of 240K Belgian resumes, acting as a highly sophisticated proxy for the PathFinder until the deterministic academic WorldCompiler is built.
*   **Missing (Design Gap):** The actual application frontend relies on hardcoded JSON files in `backend/personas/` via `pathing_engine.py` rather than running the dynamic graph algorithms locally. The true backward-chaining A* algorithm against formal academic requirements is missing.

### 1.4 Matching Agent: The Orchestration Layer
**Status: Barebones Implementation (No ML/AI)**
*   **Implemented:** `gap_engine.py` can calculate raw arithmetic deltas between a candidate's skill scores and an ideal job description (Target - Actual).
*   **Missing (Design Gap):** The "Seven Dimensions" of gap analysis are completely missing. It does not use dynamic cosine similarity vectors to recommend tasks. Instead, it uses `if/elif` statements to map specific missing skills (e.g., "C++") to hardcoded string recommendations (e.g., "Complete Custom Architecture Prompt").

### 1.5 Verification Agent (The Audit Oracle)
**Status: Mocked**
*   **Implemented:** The file `github_oracle.py` exists as a structural placeholder. The system utilizes heavy pre-computation for specific user demos ("Meenal", "Alex", "Maya", "Sophia") via static JSON persona files in `backend/personas/`. Currently, only basic testing is implemented where routes, current skills, and target roles are hardcoded to ensure the frontend demo is deterministic and smooth.
*   **Missing (Design Gap):** There is absolutely no static analysis, AST parsing, or dynamic LLM review of code repositories. The script merely uses `time.sleep()` to simulate processing time, prints fake reasoning chains to the logger, and returns a hardcoded skill vector.

### 1.6 Frontend Application (Client UI)
**Status: Fully Implemented**
*   **Implemented:** A complete React 19 / TypeScript / Vite web application residing in `src/`, utilizing Tailwind CSS v4 and Framer Motion. It includes complex UI components for rendering Subway Maps, Skill Graphs, and Profile Dashboards representing the "Optimal Path UI" generated by the AI engines.
